NgRx

Question: What is NgRx?

  • Answer:
  • reactive state management library, inspired by Redux and built on RxJS

Question: What problem does NgRx solve?

  • Answer:
  • complex state management
  • shared state across components
  • unpredictable data flow in large Angular applications

Question: When should you avoid NgRx?

  • Answer:
  • Small applications, simple forms, or when state is limited to a single component.

Question: Core Building Blocks

  • Answer: Store
    • Centralized, immutable application state.
    • Read-only from components.
    • Updated only via dispatched actions.
  • Answer: Actions
    • Describe WHAT happened → Sent to UI or Effects
    • Plain objects describing what happened.
    • It has a type and optional payload.
    • Example: loadUsers, addItem, loginSuccess.
  • Answer: Reducers
    • Pure functions that calculate new state from old state + action.
    • No side effects.
    • Pure
    • Deterministic
    • Synchronous
  • Answer: Selectors
    • Pure functions used to extract specific slices of state from the store.
    • Read-only queries to extract data from the store.
    • Improve performance via memoization.
  • Answer: Effects
    • Handle side effects (API calls, async logic).
    • Listen for actions and dispatch new actions.
    • Non-deterministic (network, time, randomness)

Question: Are Selectors memoized?

  • Answer:
  • Yes — NgRx selectors are memoized by default
  • When you create selectors using createSelector, NgRx caches the last result.

Question: Can we dispatch an action from a reducer?

  • Answer:
  • No. Reducers cannot dispatch actions.

Question: Definition of Pure Function

  • Answer:
  • Same input → same output
  • No side effects
  • Does not modify original data (immutable)

Question: Data Flow (Unidirectional)

  • Answer:
  • Component → Action → Reducer → Store → Selector → Component
  • (Action → Effect → API → Action) [for async]

Question: How to call the effect?

  • Answer:
  • Component dispatches an action → Effect listens for that action
  • loadUsers$ = createEffect(() => {})
  • Effect dispatches a new action
  • loadUsersSuccess and loadUsersFailure

Question: Reducer vs Effect

  • Answer:
  • Reducer = Calculator
  • Effect = Worker
  • Reducer: “Given old state + action → return new state”
  • Effect: “I heard this action → I’ll do work → I’ll emit another action”

Question: Quick Comparison Table

  • Answer:
  • API calls → Reducer ❌ | Effect ✅
  • Async work → Reducer ❌ | Effect ✅
  • State mutation → Reducer ❌ | Effect ❌
  • Dispatch actions → Reducer ❌ | Effect ✅
  • Pure function → Reducer ✅ | Effect ❌

Question: Do Selectors listen to actions?

  • Answer:
  • Reducer → ✅ Yes
  • Effect → ✅ Yes
  • Selector → ❌ No

Question: State mutation means?

  • Answer:
  • Changing the existing state object directly instead of creating a new state object.
  • In NgRx, state must be immutable.
  • Reducers must return a new state, not modify the old one.

Question: State (Mutation Wrong vs Immutable Correct) state updating

  • Answer:
  • on(addUser, (state, { user }) => {
  •   state.users.push(user); // ❌ mutating existing array
  •   state.count++; // ❌ modifying existing object
  •   return state; // ❌ returning same reference
  • });
  • on(addUser, (state, { user }) => ({
  •   ...state, // new object
  •   users: [...state.users, user],// new array
  •   count: state.count + 1
  • }));
  • state.user.name = 'Harish'; // wrong
  • user: { ...state.user, name: 'Harish' } // correct
  • Nested object mutation
  • state.settings.theme.dark = true;
  • settings: {
  •   ...state.settings,
  •   theme: {
  •     ...state.settings.theme,
  •     dark: true
  •   }
  • }

Question: What is state mutation?

  • Answer:
  • Directly modifying the existing state object or its nested properties

Question: What happens if state is mutated?

  • Answer:
  • We are not changing the object reference
  • Selectors may not emit
  • UI may not update
  • DevTools break

Question: Are selectors synchronous or asynchronous?

  • Answer:
  • Selectors are synchronous and pure

Question: Difference between Store and Service?

  • Answer:
  • Services manage local or feature-specific state imperatively
  • Store manages global state reactively with immutability and unidirectional data flow

Question: Explain NgRx data flow

  • Answer:
  • Component dispatches an action
  • ↓
  • Reducer updates state
  • ↓
  • Store emits new state
  • ↓
  • Component reads via selector
  • For async With API
  • Action → Effect → API → Success/Failure Action → Reducer → Store → Selector → Component
  • For async Without API
  • Action → Reducer → Store → Selector → Component

Question: Why immutability is important in NgRx?

  • Answer:
  • efficient change detection
  • selector memoization
  • predictable state transitions
  • time-travel debugging

Question: How does NgRx handle async operations?

  • Answer:
  • Uses Effects along with RxJS operators like switchMap, mergeMap, and catchError

Question: Difference between switchMap and mergeMap in Effects?

  • Answer:
  • switchMap: Cancels previous requests (used for search)
  • mergeMap: Runs multiple requests in parallel

Question: What is Store DevTools?

  • Answer:
  • Enable debugging features like action inspection, state history, and time-travel debugging

Question: How do you handle errors in NgRx?

  • Answer:
  • Effects using catchError and dispatching failure actions
  • Reducers update error state

Question: Difference between switchMap and mergeMap in Effects?

  • Answer:
  • switchMap: Cancels previous requests (used for search)
  • mergeMap: Runs multiple requests in parallel

Question: What happens if you mutate state accidentally?

  • Answer:
  • Selectors may not emit new values
  • Change detection may fail
  • DevTools debugging becomes unreliable

Question: NgRx vs Angular Signals?

  • Answer:
  • NgRx is best for large, complex, global state with async flows
  • Signals are simpler and better suited for local or medium complexity state

Question: Does NgRx prevent duplicate API calls automatically?

  • Answer:
  • No. RxJS operators control that behavior

Question: Action dispatched but API not called — why?

  • Answer:
  • Effect is not registered or ofType does not match
  • EffectsModule.forRoot([UserEffects])
  • EffectsModule.forFeature([UserEffects])
  • ofType(loadUser) // Effect listens for loadUser

Question: Why is switchMap preferred for search APIs?

  • Answer:
  • It cancels previous requests

Question: Can we use subscribe() inside an Effect?

  • Answer:
  • No. Effects must return observables

Question: OnPush + NgRx — Does it work correctly?

  • Answer:
  • When NgRx updates state:
  • Reference changes → OnPush detects → UI updates

Question: When OnPush + NgRx does NOT work

  • Answer:
  • The real issue is state mutation
  • state.users.push(user); // ❌ mutation
  • When reference is not changed

Question: Can Effects be time-traveled?

  • Answer:
  • No. Only reducers/state can be replayed

Question: Prevent duplicate API calls if data already in store

  • Answer:
  • Use withLatestFrom + selector + condition

Question: Multiple API calls on navigation

  • Answer:
  • Cause: Effect subscribed multiple times due to lazy module reloading
  • Fix: Use forFeature properly and avoid duplicate effect registration

Question: Can we store UI state in NgRx?

  • Answer:
  • YES—but selectively
  • Sidebar open/close (used across pages)
  • Theme (dark/light)
  • Logged-in user UI flags
  • Global loading spinner
  • Filters used in multiple components
  • Other are store in ngrx

Question: Trap 4

  • ❌ No action log, no replay mechanism.

Question: Is NgRx obsolete because of Signals?

  • Answer:
  • No. Signals are reactive primitives, not state architecture replacements.

  • Answer:
  • Move backward and forward through dispatched actions
  • See application state at each step
  • Replay actions deterministically

  • Answer:

One-Liner Interview Answers (Rapid-Fire)

  • NgRx: Centralized, reactive state management for Angular with immutable state
  • Store: Single source of truth for application state
  • Action: Describes what happened in the application
  • Reducer: Pure function that returns new state; no side effects
  • Effect: Handles side effects and dispatches new actions
  • Selector: Memoized, read-only state query
  • Immutability: State is never modified; new references are returned
  • State mutation: Directly changing existing state (forbidden)

  • createFeatureSelector: Extracts a feature slice from global state
  • NgRx Entity: Normalized collection management using ids and entities
  • Side effect: Any operation outside state calculation (API, router, storage)
  • Unidirectional flow: Action → Reducer → Store → Selector → View
  • DevTools: Enables action inspection and time-travel debugging
  • OnPush + NgRx: Works because of immutable state references
  • Async in NgRx: Always handled in Effects

Question: Diff between Adapter, createSelector, Entity, Module, Feature?

  • Answer: Adapter
    • What it is:
    • A helper from @ngrx/entity to manage normalized collections
    • Purpose:
    • Provides immutable CRUD operations and base selectors
    • Example:
    • createEntityAdapter<User>()
    • Think of it as:
    • DAO / Repository layer for store entities
  • Answer: createSelector
    • What it is:
    • A function to create memoized selectors
    • Purpose:
    • Derives, combines, and computes data from store state
    • Example:
    • createSelector(selectUsers, selectOrders, (u, o) => ...)
    • Think of it as:
    • Query layer (like SQL views)
  • Answer: Entity
    • What it is:
    • A domain model stored in normalized form (ids + entities)
    • Purpose:
    • Represents business objects (User, Order, Product)
    • Example:
    • EntityState<User>
    • Think of it as:
    • Database table row
  • Answer: Module
    • What it is:
    • Angular NgModule or standalone feature that groups functionality
    • Purpose:
    • Wires Store, Effects, Components, Services
    • Example:
    • StoreModule.forRoot()
    • Think of it as:
    • Application boundary / deployment unit
  • Answer: Feature
    • What it is:
    • A slice of application state with its own reducer, selectors, effects
    • Purpose:
    • Logical domain separation (Auth, Orders, Users)
    • Example:
    • StoreModule.forFeature('users', usersReducer)
    • Think of it as:
    • Bounded context in DDD
