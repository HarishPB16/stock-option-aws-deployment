📌 Phase 1: AWS Manual Deployment Guide

  1. Create an AWS account and login to AWS Console as Root User.
  2. Create an IAM Admin User with AdministratorAccess and use it for daily AWS activities.
  3. Create github-actions-user for CI/CD and GitHub Actions deployment access.
  4. Attach required IAM policies:
    • AmazonEC2ContainerRegistryPowerUser
    • AmazonS3FullAccess
  5. Generate and securely save the AWS Access Key ID and AWS Secret Access Key for GitHub integration and automated deployments.
  6. Configure GitHub repository secrets:
    • AWS_ACCESS_KEY_ID
    • AWS_SECRET_ACCESS_KEY
    • AWS_REGION
  7. Configure AWS Access and Secrets keys in Github
    • GitHub -> Settings -> Secrets and variables -> Actions

📌 Phase 2: Backend Deployment (Docker & App Runner)

  1. Create an Elastic Container Registry (ECR) repository named stock-app-backend in your selected AWS region.
  2. Copy and save the generated ECR Repository URI for Docker image deployment.
  3. Create the GitHub Actions workflow file: .github/workflows/backend-deploy.yml for automated CI/CD deployment.
  4. backend-deploy.yml Workflow:
    • Trigger deployment when code is pushed to main branch
    • Run only when backend files are changed
    • Checkout latest source code from GitHub
    • Configure AWS credentials using secrets
    • Login to Amazon ECR
    • Build Docker image
    • Tag Docker image with ECR URI and latest tag
    • Push Docker image to ECR
    • Prepare backend image for AWS App Runner deployment
  5. Create an AWS App Runner service using the Docker image from Amazon ECR.
  6. Add required environment variables such as:
    • MONGO_URI
    • OPENAI_API_KEY
  7. Goto the MongoDB Atlas Network Access to allow: 0.0.0.0/0 for App Runner connectivity.
  8. Commit and push the workflow file to GitHub and verify deployment from the GitHub Actions tab.

📌 Phase 3: Frontend Deployment (S3 & CloudFront)

  1. Update the Angular production environment file with the AWS App Runner Backend URL and run: npm run build to generate the production build.
  2. Create an AWS S3 Bucket, enable Static Website Hosting, and configure: index.html as both Index and Error document.
  3. Configure the S3 Bucket Policy to allow public read access for frontend hosting.
  4. Create: .github/workflows/frontend-deploy.yml to automatically build and deploy the Angular application to S3.
  5. frontend-deploy.yml Workflow
    • Trigger deployment when code is pushed to main branch
    • Run only when frontend files are changed
    • Checkout latest source code from GitHub
    • Setup Node.js environment
    • Install frontend dependencies using npm ci
    • Build Angular frontend application
    • Configure AWS credentials using GitHub Secrets
    • Upload frontend build files to AWS S3 bucket
    • Remove old files from S3 using --delete option
    • Optionally clear CloudFront cache after deployment
  6. Create an AWS CloudFront Distribution for HTTPS support, caching, and faster global delivery.
  7. Configure CloudFront Origin Access Control (OAC), redirect HTTP to HTTPS, and set: index.html as the default root object.
  8. Configure custom 403 and 404 error responses in CloudFront to support Angular routing.
  9. Update the backend FRONTEND_URL environment variable in AWS App Runner with the CloudFront domain URL and restart the service.

📌 Phase 4: Final Polish

  1. CORS configuration : Update the backend FRONTEND_URL environment variable in AWS App Runner using your live CloudFront URL.
  2. Restart the AWS App Runner Service to apply the updated CORS configuration.
  3. Verify frontend and backend connectivity using the live CloudFront and App Runner URLs.

👉 Your application is now fully separated and deployed on AWS!

GitHub Actions Workflow

name: Deploy Backend to ECR

on:

  push:

    branches: ["main"]

    # Triggers on pushing to the main branch

    paths:

      - "backend/**"

      # Only triggers if files in backend change

jobs:

  build-and-push:

    runs-on: ubuntu-latest

    steps:

      - name: Checkout Code

        uses: actions/checkout@v3

      - name: Configure AWS credentials

        uses: aws-actions/configure-aws-credentials@v2

        with:

          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}

          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}

          aws-region: ${{ secrets.AWS_REGION }}

      - name: Login to Amazon ECR

        id: login-ecr

        uses: aws-actions/amazon-ecr-login@v1

      - name: Build, tag, and push image to Amazon ECR

        env:

          ECR_REGISTRY: ${{ steps.login-ecr.outputs.registry }}

          ECR_REPOSITORY: stock-app-backend

          IMAGE_TAG: latest

        run: |

          cd backend

          docker build -t $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG .

          docker push $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG
  1. Commit and push this file to GitHub

📌 Common CI/CD & AWS Deployment Issues

  1. YAML Indentation Errors
    • Mistake: Wrong spacing or tabs inside GitHub workflow YAML files.
    • Symptom: Invalid workflow file or YAML syntax errors.
  2. Wrong AWS Region
    • Mistake: AWS services created in different regions than configured in GitHub Secrets.
    • Symptom: Repository not found or bucket does not exist.
  3. Wrong AWS Credentials
    • Mistake: Invalid or expired AWS Access Key and Secret Key.
    • Symptom: InvalidClientTokenId or SignatureDoesNotMatch errors.
  4. Missing IAM Permissions
    • Mistake: Required IAM policies are not attached.
    • Symptom: AccessDeniedException or ECR/S3 permission failures.
  5. Dockerfile or Build Context Issues
    • Mistake: Missing Dockerfile or incorrect build directory.
    • Symptom: Docker build fails with Dockerfile not found.
  6. Wrong ECR Repository Name
    • Mistake: Repository name mismatch between AWS and GitHub Actions.
    • Symptom: Repository does not exist or name unknown errors.
  7. App Runner Port Mismatch
    • Mistake: App Runner port differs from Node.js application port.
    • Symptom: Health check failed or deployment rollback.
  8. MongoDB Network Access Issues
    • Mistake: MongoDB Atlas whitelist not configured for App Runner.
    • Symptom: MongoNetworkError or IP not whitelisted.
  9. Environment Variable Issues
    • Mistake: Missing or incorrect environment variables.
    • Symptom: Backend crashes, missing API keys, or API failures.
  10. S3 Bucket Policy or Public Access Issues
    • Mistake: Public access blocked or bucket policy missing.
    • Symptom: 403 Forbidden error on website.
  11. Angular Routing Configuration Missing
    • Mistake: index.html not configured for error handling.
    • Symptom: 404 Access Denied on page refresh.
  12. CloudFront Cache Issues
    • Mistake: Cached old frontend files after deployment.
    • Symptom: Old UI or outdated API URLs still visible.
  13. Wrong CORS FRONTEND_URL
    • Mistake: Backend still points to localhost instead of CloudFront.
    • Symptom: Browser blocks API requests due to CORS policy.
  14. Wrong Angular Build Folder Uploaded
    • Mistake: Incorrect dist folder uploaded to S3.
    • Symptom: Blank page, missing assets, or broken UI.
  15. GitHub Workflow Path Issues
    • Mistake: Incorrect paths syntax in GitHub Actions.
    • Symptom: Workflow never triggers after push.
  16. GitHub Secret Name Mismatch
    • Mistake: Secret name differs from YAML reference.
    • Symptom: Deployment variables become empty or null.
  17. Missing Frontend Build Artifacts
    • Mistake: Frontend build not generated before deployment.
    • Symptom: Empty website or missing uploaded files.

📌 How to Debug Fast

  1. Check GitHub Actions Logs
    • Go to GitHub Repository → Actions → Failed Workflow → Logs
    • Identify YAML errors, Docker failures, AWS authentication issues, or missing secrets.
  2. Check AWS App Runner Logs
    • Open AWS Console → App Runner → Logs
    • Identify port mismatch, MongoDB errors, missing variables, or backend crashes.
  3. Verify GitHub Secrets
    • Validate AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_REGION, and AWS_S3_BUCKET.
    • Ensure secret names exactly match GitHub workflow references.
  4. Validate Environment Variables
    • Check MONGO_URI, FRONTEND_URL, OPENAI_API_KEY, and other required variables.
    • Verify frontend and backend URLs are correct.
  5. Check Browser Developer Console
    • Press F12 → Console & Network tab.
    • Identify CORS issues, API failures, missing assets, or runtime errors.
  6. Test Backend APIs Directly
    • Test App Runner APIs using Browser or Postman.
    • Verify backend availability, routing, and database connectivity.
  7. Verify AWS Region Configuration
    • Ensure ECR, S3, App Runner, and GitHub Actions use the same AWS region.
    • Prevent repository lookup and authentication issues.
  8. Check Docker Build Locally
    • Run docker build locally before pushing changes.
    • Identify Dockerfile, dependency, or build context problems.
  9. Validate S3 Static Hosting Settings
    • Ensure index.html is configured correctly.
    • Verify bucket policy and public access settings.
  10. Check CloudFront Distribution
    • Ensure distribution status is Enabled.
    • Verify cache invalidation after frontend deployment.
  11. Verify Angular Production Build
    • Run npm run build locally.
    • Ensure dist/frontend/browser exists without build errors.
  12. Validate GitHub Workflow Paths
    • Verify frontend/** and backend/** paths configuration.
    • Ensure workflows trigger correctly after push.
  13. Verify ECR Repository Configuration
    • Check repository name, region, and visibility.
    • Prevent Docker push and repository lookup failures.
  14. Verify App Runner Port Configuration
    • Ensure App Runner port matches app.listen(3000).
    • Prevent health check and deployment failures.
  15. Test S3 Website Endpoint Directly
    • Open the S3 website endpoint URL directly.
    • Verify frontend hosting and bucket configuration.

📌 AWS Deployment Interview Questions & Answers

  1. Explain your end-to-end deployment architecture.

    Angular frontend hosted on AWS S3 + CloudFront, Node.js backend deployed using Docker on AWS App Runner, Docker images stored in AWS ECR, MongoDB Atlas used as database, and GitHub Actions used for CI/CD automation.

  2. Why did you use Docker for backend deployment?

    Docker provides environment consistency, portability, dependency management, simplified deployment, faster scaling, and eliminates environment mismatch issues.

  3. Why did you choose AWS App Runner instead of EC2?

    App Runner is fully managed, supports automatic scaling, HTTPS, Docker deployment, and reduces server management overhead compared to EC2.

  4. Explain your GitHub Actions CI/CD pipeline flow.

    Backend pipeline builds Docker image and pushes to ECR, while frontend pipeline builds Angular app and deploys files to S3 automatically on every push to the main branch.

  5. How does GitHub authenticate with AWS?

    GitHub authenticates using IAM Access Key and Secret Key stored securely inside GitHub Secrets using aws-actions/configure-aws-credentials.

  6. What are GitHub Secrets?

    GitHub Secrets securely store sensitive values such as AWS credentials, API keys, and deployment configuration variables used in CI/CD workflows.

  7. What is AWS ECR?

    AWS ECR is a private Docker image registry used to store, manage, and version Docker container images for deployment.

  8. Why did you use S3 + CloudFront for Angular hosting?

    S3 hosts static frontend files while CloudFront provides HTTPS, CDN caching, faster delivery, scalability, and improved performance.

  9. Why is CloudFront needed if S3 already hosts the frontend?

    CloudFront adds HTTPS support, global CDN acceleration, caching, better security, and faster frontend loading compared to S3 static hosting alone.

  10. How did you solve Angular page refresh issues?

    Configured index.html as Error Document in S3 and added custom 403 and 404 CloudFront error responses redirecting to index.html.

  11. What is CORS and how did you configure it?

    CORS controls cross-origin browser requests. Configured FRONTEND_URL in backend environment variables to allow trusted frontend domains.

  12. How do you debug deployment failures quickly?

    I check GitHub Actions logs, App Runner logs, environment variables, browser console, backend APIs, and AWS service configurations to isolate issues quickly.

  13. How do you manage environment variables securely?

    Frontend uses Angular environment files, backend uses App Runner environment variables, and CI/CD secrets are stored securely in GitHub Secrets.

  14. How does App Runner know a new Docker image is available?

    App Runner continuously monitors ECR repositories and automatically redeploys when GitHub Actions pushes a new Docker image.

  15. What production issues did you face during deployment?

    Common issues included YAML errors, wrong AWS region, IAM permission issues, MongoDB whitelist problems, CORS errors, CloudFront caching, wrong ports, and missing environment variables.

📌 Advanced CI/CD, AWS & DevOps Interview Questions

  1. What is Infrastructure as Code (IaC)?

    Infrastructure as Code allows infrastructure provisioning using tools like Terraform or AWS CDK instead of manual AWS console configuration. It improves consistency, automation, version control, and environment replication.

  2. How do you secure CI/CD pipelines?

    Secrets should never be hardcoded. GitHub Secrets or OIDC authentication should be used to securely authenticate GitHub Actions with AWS without exposing credentials.

  3. How do you optimize GitHub Actions build time?

    Docker Layer Caching and GitHub Actions cache are used to reduce rebuild time by caching dependencies and Docker layers.

  4. How do you manage multiple environments like staging and production?

    GitHub Environments and separate AWS credentials are used for staging and production deployments with conditional workflow execution.

  5. Why use App Runner instead of EC2?

    App Runner is serverless and fully managed, while EC2 requires manual server maintenance, scaling, patching, and load balancer management.

  6. How do you debug App Runner health check failures?

    Verify application port configuration, environment variables, database connectivity, and startup timing issues.

  7. What is stateless architecture in containers?

    Containers should not store persistent data locally because multiple instances can be created dynamically. External services like S3, Redis, or MongoDB should store data.

  8. How do you improve container security?

    Enable Amazon ECR Enhanced Scanning or Amazon Inspector to scan Docker images for security vulnerabilities before deployment.

  9. Why does Angular routing fail on refresh?

    S3 cannot understand Angular client-side routes. CloudFront custom error responses redirect 403 and 404 errors back to index.html for Angular routing.

  10. What is CloudFront cache invalidation?

    Cache invalidation removes old cached frontend files from CloudFront edge locations to ensure users receive the latest deployment.

  11. What is OAC (Origin Access Control)?

    OAC allows CloudFront to securely access private S3 buckets without exposing the bucket publicly to users.

  12. How does CloudFront reduce latency?

    CloudFront caches frontend files in global edge locations so users download content from nearby servers instead of the origin server.

  13. How does CORS work?

    Browsers send a preflight OPTIONS request to validate allowed origins. The backend must return proper Access-Control-Allow-Origin headers.

  14. How would you improve database security?

    Use App Runner VPC Connector, private subnets, VPC peering, or AWS DocumentDB to avoid exposing database traffic publicly.

  15. How do you optimize AWS infrastructure cost?

    Monitor CloudFront, App Runner, and data transfer costs using AWS Cost Explorer and configure CloudWatch billing alarms for budget tracking.

📌 Using package.json Version as Docker Image Tag

This setup automatically reads the application version from package.json and uses it as the Docker image tag during GitHub Actions deployment.

1. Define Version inside package.json 

{
  "name": "backend",
  "version": "1.2.0"
}

2. Read Version from package.json

Run the following command inside Linux or Mac terminal: 

node -p "require('./package.json').version"

Output: 

1.2.0

3. Store Version into Variable 

VERSION=$(node -p "require('./package.json').version")

4. Build Docker Image using Version Tag 

docker build -t node-backend:$VERSION .

Use Version Tag inside GitHub Actions Workflow 

run: |
  cd backend

  VERSION=$(node -p "require('./package.json').version")

  docker build \
    -t $ECR_REGISTRY/$ECR_REPOSITORY:$VERSION \

  docker push $ECR_REGISTRY/$ECR_REPOSITORY:$VERSION

Result

  • Docker image tagged with: 
    1.2.0

📌 Docker, Containers, EC2, ECR & GitHub Runner Concepts

1. What is Docker?

Docker is a platform used to package:

  • Application code
  • Dependencies
  • Libraries
  • Runtime
  • Operating system components

together so the application runs the same everywhere.

2. What is Docker Image?

Docker Image is a packaged snapshot of the application.

Created using: 

docker build

3. What is Container?

Container is a running instance of Docker image.

Simple Flow: 

Dockerfile
    ↓
Docker Image
    ↓
Container

4. Difference between Docker Image and Container

  • Docker Image → Static packaged file
  • Container → Running live application

5. Is Container Running Under EC2?

Technically yes internally.

But AWS manages the infrastructure automatically.

In App Runner, AWS internally uses compute infrastructure like:

  • EC2
  • Fargate
  • Container runtime

But for developers:

👉 App Runner abstracts infrastructure management.

Meaning:

  • No server management
  • No OS patching
  • No manual scaling

6. What is EC2?

EC2 stands for: 

Elastic Compute Cloud

EC2 is a virtual machine/server running in AWS cloud.

It behaves like a remote Linux computer.

7. What is ECR?

ECR stands for: 

Amazon Elastic Container Registry

ECR is a Docker image storage repository.

It stores: 

Docker Images

NOT running containers.

8. What is GitHub Runner?

GitHub Runner is a temporary Linux VM created by GitHub Actions.

Used to:

  • Run workflows
  • Build Docker images
  • Run tests
  • Deploy code

9. How Many Docker Images are Stored by GitHub and AWS?

GitHub:

  • Normally GitHub does not store Docker images
  • Unless using: 
    GitHub Container Registry (GHCR)

AWS ECR:

  • Stores multiple Docker image versions/tags
  • Example: 
    backend:v1
backend:v2
backend:latest
backend:prod
  • Many Docker images can be stored depending on AWS storage limits.

📌 Top 15 AWS + GitHub Pipeline Security Interview Questions and Answers

1. Why should we avoid using AWS Root Account for deployments?

The AWS Root Account has unrestricted access to all AWS services and billing.

Security risks:

  • Full account compromise if leaked
  • No granular permissions
  • Difficult auditing

Best practice:

  • Create IAM users/roles
  • Enable MFA on root account
  • Use root only for critical account-level tasks

2. Why do we create a separate IAM user for GitHub Actions?

GitHub Actions requires programmatic access to AWS.

Instead of using admin credentials:

  • Create dedicated IAM user or IAM role
  • Assign only required permissions

Example:

  • ECR push access
  • S3 deployment access

This follows: 

Principle of Least Privilege (PoLP)

3. Why is storing AWS keys directly in GitHub workflow dangerous?

Hardcoding keys in YAML files is a major security issue because:

  • Keys become visible in repository history
  • Anyone with repo access can steal them
  • Keys may leak publicly accidentally

Correct approach: 

Store secrets in GitHub Secrets

Example: 

${{ secrets.AWS_ACCESS_KEY_ID }}

4. What are GitHub Secrets?

GitHub Secrets are encrypted environment variables used securely inside GitHub Actions.

Used for:

  • AWS keys
  • DB passwords
  • API tokens

Advantages:

  • Encrypted at rest
  • Hidden from logs
  • Repository-level protection

5. Why is IAM Role better than Access Keys for GitHub pipelines?

Access keys are static credentials.

Problems:

  • Can leak
  • Require manual rotation
  • Long-term security risk

IAM Role with OIDC:

  • Temporary credentials
  • No secret storage
  • Automatic token expiration
  • Better compliance

Modern AWS pipelines prefer: 

GitHub OIDC + IAM Role

6. What is OIDC in GitHub Actions?

OIDC (OpenID Connect) allows GitHub to authenticate directly with AWS without storing AWS secrets.

Flow:

  1. GitHub generates temporary token
  2. AWS validates GitHub identity
  3. AWS provides temporary credentials

Benefits:

  • No static secrets
  • Improved security
  • Temporary access

7. Why should S3 buckets not be public?

Public S3 buckets expose data to the internet.

Risks:

  • Data leakage
  • Sensitive file exposure
  • Malware hosting
  • Compliance violations

Best practice:

  • Keep buckets private
  • Use CloudFront + Origin Access Control (OAC)

8. Why is CloudFront more secure than direct S3 access?

CloudFront improves security by:

  • Providing HTTPS
  • Hiding S3 bucket
  • Preventing direct access
  • Adding caching
  • Supporting WAF

Best practice: 

Allow only CloudFront to access S3

9. Why is 0.0.0.0/0 dangerous in MongoDB Atlas?

0.0.0.0/0 allows access from anywhere on the internet.

Risks:

  • Brute force attacks
  • Unauthorized access
  • Increased attack surface

Better solutions:

  • AWS VPC peering
  • Specific IP whitelisting
  • Private networking

10. Why should secrets not be stored inside .env files in GitHub?

If .env files are committed:

  • Secrets leak permanently
  • Git history retains them
  • Attackers can access infrastructure

Best practice:

  • Use GitHub Secrets
  • Use AWS Secrets Manager
  • Use Parameter Store

11. What is the Principle of Least Privilege?

Users/services should receive only minimum permissions required.

Bad example: 

AdministratorAccess

Good example:

  • ECR push only
  • S3 sync only

Benefits:

  • Reduces blast radius
  • Prevents accidental damage
  • Improves compliance

12. Why enable MFA on AWS accounts?

MFA adds second-layer authentication.

Even if password leaks:

Attacker still needs OTP/device.

Critical for:

  • Root account
  • IAM admin users

13. What security mistake commonly happens in CI/CD pipelines?

Common mistakes:

  • Hardcoded secrets
  • Excessive IAM permissions
  • Public S3 buckets
  • No HTTPS
  • Exposed logs
  • No secret rotation

Realistic examples:

  • MongoDB opened to all IPs
  • Missing environment variables
  • Backend failed due to DB connection issue

14. How does AWS know GitHub code changed and deployment is needed?

GitHub Actions listens for Git events.

Example: 

on:
  push:
    branches:
      - main

Flow:

  1. Developer pushes code
  2. GitHub triggers workflow
  3. Workflow builds app
  4. Docker image pushed to ECR
  5. App Runner auto-deploys new image

15. Why is bucket versioning important?

S3 Versioning keeps multiple versions of files.

Benefits:

  • Rollback capability
  • Protection from accidental deletion
  • Recovery from bad deployments
  • Ransomware protection

Example: 

Bad Angular build deployed

Restore previous index.html

Very important for production CI/CD.

Dangerous in team environments.
