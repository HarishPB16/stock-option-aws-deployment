📌 MBD Project Explanation

MBD (Major Breakdown) is a system used in car manufacturing to manage critical production issues.

Business Context

  • Manufacturing has dependent units: Assembly, Paint Shop, Body Shop
  • If one unit fails → entire production line is impacted
  • Issue > 10 mins → classified as Major Breakdown

Objective

Build a system to coordinate multiple teams and resolve breakdowns quickly, minimizing production downtime.

Roles & Responsibilities

  • Job Coordinator: Creates MBD, assigns tasks, closes ticket
  • Engineer: Identifies root cause and fixes issue
  • Safety Team: Performs safety validation
  • Shop Coordinator: Confirms system is operational

Workflow

  • Issue occurs
  • Engineer fixes the issue
  • Safety team validates
  • Shop coordinator confirms operations
  • Job coordinator closes the MBD

System Capabilities

  • Role-based UI screens
  • End-to-end lifecycle tracking
  • Real-time coordination between teams

👉 Goal: Reduce downtime and ensure faster resolution of production breakdowns

Technical Overview

  • Frontend: Angular (SPA)
  • Backend: AWS (Lambda / ECS based)
  • Storage: S3 + Aurora PostgreSQL
  • Delivery: CloudFront

📌 Architecture (End-to-End Flow)

1. User Access Layer

  • User: TMNA internal user
  • Access: Zscaler Private Access (ZPA)
  • Auth: Azure AD (SSO)
  • Client: Web Browser

👉 Secure access + Single Sign-On (SSO)

2. Entry Point (Internet → AWS)

  • HTTPS (Port 443)
  • Route 53 (DNS resolution)

👉 Route 53 = Domain routing layer

3. CDN + Security

  • CloudFront (CDN)
  • WAF (Firewall)
  • SSL via Certificate Manager
  • S3 (Logs storage)

👉 Fast + Secure delivery

4. Frontend Hosting

  • S3 (Angular SPA)
  • Served via CloudFront

👉 Serverless frontend

5. API Layer

  • API Gateway
  • Routing + Security + Rate limiting

6. Backend

  • Lambda (Business logic)
  • Runs on demand (event-based)

👉 Auto scaling, no servers

7. Database

  • Aurora PostgreSQL Serverless
  • Private subnet (not public)

8. File Storage

  • S3 (Images, reports, attachments)

9. Monitoring & Security

  • CloudWatch (logs)
  • Datadog (monitoring)
  • Secrets Manager (credentials)

10. Network Design

  • VPC + Private Subnets
  • Backend & DB are NOT public
  • Only API Gateway exposed

11. Why Serverless

  • Cost efficient (scale to zero)
  • No infra management
  • High scalability

12. Integration

  • Maximo (work orders)
  • Future: EDC, Nexus Broker

13. End-to-End Flow

  • User login (Azure AD)
  • Route 53 → CloudFront
  • Angular UI loads
  • API Gateway → Lambda
  • Lambda → Aurora + S3
  • Logs → CloudWatch

👉 Cloud-Native + Serverless + Secure + Cost-Optimized

Authentication Flow (Azure AD + MSAL)

  1. Install MSAL packages
  2. Configure MSAL
  3. Provide MSAL
  4. Login
  5. Get Token
  6. Attach Token
  7. Logout
  8. Backend validates token

1. Install

npm install @azure/msal-browser @azure/msal-angular

2. MSAL Config

const msalInstance = new PublicClientApplication({
  auth: {
    clientId: 'CLIENT_ID',
    authority: 'https://login.microsoftonline.com/TENANT_ID',
    redirectUri: 'http://localhost:4200'
  }
});

3. Provide MSAL

providers: [
  provideMsal(msalInstance)
]

4. Login

this.msal.loginPopup({ scopes: ['user.read'] })

5. Get Token

this.msal.acquireTokenSilent({ scopes: ['user.read'] })

6. Attach Token

Authorization: Bearer ${token}

7. Logout

this.msal.logoutPopup()

8. Backend Validation

jwt.decode(token, key, algorithms=["RS256"])

Lambda Authorization Approaches

✅ Option 1: Separate Lambda Authorizer (Recommended)

Client → API Gateway → Authorizer → Business Lambda
  • Centralized authentication logic
  • Reusable across APIs
  • Clean separation of concerns
  • Easy to maintain
Authorizer Example
token = event["authorizationToken"]

if token == "allow":
  return {
    "principalId": "user123",
    "context": {"role": "admin"}
  }
else:
  raise Exception("Unauthorized")
Business Lambda
role = event["requestContext"]["authorizer"]["role"]

if role != "admin":
  return {"statusCode": 403}

⚡ Option 2: Inline Authorization (Same Lambda)

Client → API Gateway → Lambda (auth + logic)
  • Simple implementation
  • Good for small apps / POC
Example
token = event["headers"].get("Authorization")

if token != "allow":
  return {"statusCode": 403}

return {"statusCode": 200}
  • ❌ Duplicate logic
  • ❌ Hard to maintain
  • ❌ Security risk

🧩 Option 3: Shared Layer

  • Use Lambda Layer for shared auth code
  • Reusable across Lambdas

Decision Guide

  • Small App → Inline Auth
  • Medium App → Shared Layer
  • Production → Separate Authorizer

Key Insight

Authorizer runs BEFORE Business Lambda
If Deny → Lambda is NOT executed

Interview Answer

In production, we use a separate Lambda Authorizer with API Gateway to centralize authentication and ensure consistent security across APIs.

📌 AWS Lambda with API Gateway: Step-by-Step Deployment Guide via GitHub Actions

This guide explains how to deploy a Python-based AWS Lambda function integrated with API Gateway using GitHub Actions CI/CD pipeline.

PART 1: PREREQUISITES

  1. AWS Account with IAM access
  2. GitHub Repository containing Python code
  3. AWS CLI installed locally (optional)

PART 2: PROJECT STRUCTURE

.
├── src/
│   └── lambda_function.py
│
├── requirements.txt
├── template.yaml
│
└── .github/
    └── workflows/
        └── deploy.yml

PART 3: AWS SAM TEMPLATE (template.yaml)

AWS SAM simplifies Lambda + API Gateway infrastructure creation. 

AWSTemplateFormatVersion: '2010-09-09'

Transform: AWS::Serverless-2016-10-31

Resources:

  MyApiFunction:

    Type: AWS::Serverless::Function

    Properties:

      Handler: lambda_function.lambda_handler

      Runtime: python3.9

      CodeUri: src/

      Events:

        HelloWorld:

          Type: Api

          Properties:

            Path: /hello

            Method: get

PART 4: CONFIGURE AWS PERMISSIONS

  1. Create IAM User in AWS Console
  2. Attach permissions:
    • AdministratorAccess (easy setup)
    • OR specific Lambda/API Gateway permissions
  3. Generate Access Key ID and Secret Access Key
  4. Open GitHub Repository: 
    Settings → Secrets and variables → Actions
  5. Add GitHub Secrets: 
    AWS_ACCESS_KEY_ID
AWS_SECRET_ACCESS_KEY
AWS_REGION

PART 5: GITHUB ACTIONS WORKFLOW

File Location: 

.github/workflows/deploy.yml

This workflow deploys Lambda automatically whenever code is pushed to the main branch. 

name: Deploy to AWS Lambda

on:
  push:
    branches:
      - main

jobs:

  deploy:

    runs-on: ubuntu-latest

    steps:

      - name: Checkout code
        uses: actions/checkout@v3

      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.9'

      - name: Install AWS SAM CLI
        run: |
          pip install aws-sam-cli

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v2
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ${{ secrets.AWS_REGION }}

      - name: SAM Build
        run: sam build

      - name: SAM Deploy
        run: |
          sam deploy \
            --no-confirm-changeset \
            --no-fail-on-empty-changeset \
            --stack-name my-lambda-stack \
            --resolve-s3 \
            --capabilities CAPABILITY_IAM

PART 6: DEPLOYMENT STEPS SUMMARY

  1. Commit and push source code to GitHub
  2. GitHub Actions workflow automatically starts
  3. Open GitHub → Actions tab to monitor deployment
  4. CloudFormation creates stack: 
    my-lambda-stack
  5. Open AWS CloudFormation → Outputs tab
  6. Copy generated API Gateway URL
  7. Test endpoint: 
https://your-api-id.execute-api.us-east-1.amazonaws.com/Prod/hello

📌 PART 5: GitHub Actions Workflow (.github/workflows/deploy.yml)

Multi Environment Deployment

  • develop → dev environment
  • staging → stage environment
  • main → production environment

Workflow File

name: Deploy to AWS Lambda

on:
  push:
    branches:
      - develop
      - staging
      - main

jobs:

  deploy:

    runs-on: ubuntu-latest

    steps:

      - name: Checkout code
        uses: actions/checkout@v3

      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.12'

      - name: Install AWS SAM CLI
        run: |
          pip install aws-sam-cli

      - name: Set Environment Variables
        run: |

          if [ "${GITHUB_REF##*/}" = "develop" ]; then

            echo "STACK_NAME=my-lambda-dev" >> $GITHUB_ENV

            echo "AWS_ACCESS_KEY_ID=${{ secrets.DEV_AWS_ACCESS_KEY_ID }}" >> $GITHUB_ENV

            echo "AWS_SECRET_ACCESS_KEY=${{ secrets.DEV_AWS_SECRET_ACCESS_KEY }}" >> $GITHUB_ENV

          elif [ "${GITHUB_REF##*/}" = "staging" ]; then

            echo "STACK_NAME=my-lambda-stage" >> $GITHUB_ENV

            echo "AWS_ACCESS_KEY_ID=${{ secrets.STAGE_AWS_ACCESS_KEY_ID }}" >> $GITHUB_ENV

            echo "AWS_SECRET_ACCESS_KEY=${{ secrets.STAGE_AWS_SECRET_ACCESS_KEY }}" >> $GITHUB_ENV

          elif [ "${GITHUB_REF##*/}" = "main" ]; then

            echo "STACK_NAME=my-lambda-prod" >> $GITHUB_ENV

            echo "AWS_ACCESS_KEY_ID=${{ secrets.PROD_AWS_ACCESS_KEY_ID }}" >> $GITHUB_ENV

            echo "AWS_SECRET_ACCESS_KEY=${{ secrets.PROD_AWS_SECRET_ACCESS_KEY }}" >> $GITHUB_ENV

          fi

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v2
        with:
          aws-access-key-id: ${{ env.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ env.AWS_SECRET_ACCESS_KEY }}
          aws-region: us-east-1

      - name: SAM Build
        run: sam build

📌 What is ${GITHUB_REF##*/} ?

It is a Bash parameter expansion expression.

Purpose

Extract only the branch or tag name from the full Git reference.

Example

GitHub automatically provides: 

GITHUB_REF=refs/heads/main

Expression: 

${GITHUB_REF##*/}

Output: 

main

How It Works

##*/
  • ## → remove longest matching pattern from beginning
  • */ → match everything until last "/"

Remaining value: 

main

Examples Table

GITHUB_REF Result
refs/heads/develop develop
refs/heads/staging staging
refs/heads/main main
refs/tags/v1.0 v1.0

Real Usage in GitHub Actions

if [ "${GITHUB_REF##*/}" = "main" ]; then
  echo "Production Deployment"
fi

CI/CD Flow

Git Push
    ↓
GitHub sets GITHUB_REF
    ↓
Workflow reads branch name
    ↓
Environment selected
    ↓
Deployment starts

One-line Interview Answer

${GITHUB_REF##*/} extracts the branch or tag name from the full Git reference inside GitHub Actions workflows.
