📌 MBD Project Explanation

MBD (Major Breakdown) is a system used in car manufacturing to manage critical production issues.

Business Context

  • Manufacturing has dependent units: Assembly, Paint Shop, Body Shop
  • If one unit fails → entire production line is impacted
  • Issue > 10 mins → classified as Major Breakdown

Objective

Build a system to coordinate multiple teams and resolve breakdowns quickly, minimizing production downtime.

Roles & Responsibilities

  • Job Coordinator: Creates MBD, assigns tasks, closes ticket
  • Engineer: Identifies root cause and fixes issue
  • Safety Team: Performs safety validation
  • Shop Coordinator: Confirms system is operational

Workflow

  • Issue occurs
  • Engineer fixes the issue
  • Safety team validates
  • Shop coordinator confirms operations
  • Job coordinator closes the MBD

System Capabilities

  • Role-based UI screens
  • End-to-end lifecycle tracking
  • Real-time coordination between teams

👉 Goal: Reduce downtime and ensure faster resolution of production breakdowns

Technical Overview

  • Frontend: Angular (SPA)
  • Backend: AWS (Lambda / ECS based)
  • Storage: S3 + Aurora PostgreSQL
  • Delivery: CloudFront

  1. Install MSAL packages
  2. Configure MSAL
  3. Provide MSAL
  4. Login
  5. Get Token
  6. Attach Token
  7. Logout
  8. Backend validates token

1. Install

npm install @azure/msal-browser @azure/msal-angular

2. MSAL Config

const msalInstance = new PublicClientApplication({
  auth: {
    clientId: 'CLIENT_ID',
    authority: 'https://login.microsoftonline.com/TENANT_ID',
    redirectUri: 'http://localhost:4200'
  }
});

3. Provide MSAL

providers: [
  provideMsal(msalInstance)
]

4. Login

this.msal.loginPopup({ scopes: ['user.read'] })

5. Get Token

6. Attach Token

Authorization: Bearer ${token}

7. Logout

this.msal.logoutPopup()

8. Backend Validation

jwt.decode(token, key, algorithms=["RS256"])

Lambda Authorization Approaches

✅ Option 1: Separate Lambda Authorizer (Recommended)

Client → API Gateway → Authorizer → Business Lambda
  • Centralized authentication logic
  • Reusable across APIs
  • Clean separation of concerns
  • Easy to maintain
Authorizer Example
token = event["authorizationToken"]

if token == "allow":
  return {
    "principalId": "user123",
    "context": {"role": "admin"}
  }
else:
  raise Exception("Unauthorized")
Business Lambda
role = event["requestContext"]["authorizer"]["role"]

if role != "admin":
  return {"statusCode": 403}

⚡ Option 2: Inline Authorization (Same Lambda)

Client → API Gateway → Lambda (auth + logic)
  • Simple implementation
  • Good for small apps / POC
Example
token = event["headers"].get("Authorization")

if token != "allow":
  return {"statusCode": 403}

return {"statusCode": 200}
  • ❌ Duplicate logic
  • ❌ Hard to maintain
  • ❌ Security risk

🧩 Option 3: Shared Layer

  • Use Lambda Layer for shared auth code
  • Reusable across Lambdas

Decision Guide

  • Small App → Inline Auth
  • Medium App → Shared Layer
  • Production → Separate Authorizer

Key Insight

Authorizer runs BEFORE Business Lambda
If Deny → Lambda is NOT executed

Interview Answer

In production, we use a separate Lambda Authorizer with API Gateway to centralize authentication and ensure consistent security across APIs.

AWS API Gateway + Lambda Authorizer Setup

0. Assumptions

  • auth-lambda (authorizer)
  • business-lambda (main API)
  • Same AWS region

1. Create API

API Gateway → Create API → HTTP API → Build
Name: my-secure-api

2. Add Integration

Integration type: Lambda
Select: business-lambda

3. Define Routes

GET /users → attach business-lambda

4. Create Stage

Stage: prod
Auto-deploy: ON

5. Create Authorizer

Name: my-lambda-authorizer
Type: Lambda
Function: auth-lambda
Payload: 2.0
Identity: $request.header.Authorization

6. Attach Authorizer

Route: GET /users → attach authorizer

7. Permissions

Principal: apigateway.amazonaws.com
Action: lambda:InvokeFunction

8. Authorizer Lambda

def lambda_handler(event, context):
  token = event["headers"].get("authorization", "")
  if token != "valid-token":
    return {"isAuthorized": False}

  return {
    "isAuthorized": True,
    "context": {"userId": "123", "role": "admin"}
  }

9. Business Lambda

def lambda_handler(event, context):
  ctx = event["requestContext"]["authorizer"]["lambda"]
  role = ctx.get("role")

  if role != "admin":
    return {"statusCode": 403, "body": "Forbidden"}

  return {"statusCode": 200, "body": "Success"}

10. Test

curl -X GET https://api/prod/users \
-H "Authorization: valid-token"

Client → API Gateway → Authorizer → Business Lambda

12. Common Mistakes

  • Authorizer not attached
  • Wrong header (Authorization)
  • No Lambda permission
  • Wrong response format

13. Production Tips

  • Use JWT (RS256)
  • Validate issuer & expiry
  • Pass minimal context

AWS Lambda Authorization Flow (Simple Steps)

  1. Create API in AWS API Gateway
  2. Define routes (e.g., /users)
  3. Attach Business Lambda as integration
  4. Create Lambda Authorizer
  5. Read Authorization header
  6. Validate JWT / Token
  7. Allow or Deny request
  8. Attach Authorizer to routes
  9. Enable authorization for APIs
  10. Pass user context to backend
  11. Business Lambda runs only if authorized

Flow

Client → API Gateway → Authorizer → Business Lambda

Key Points

  • Authorizer validates token
  • User info passed in requestContext
  • Unauthorized requests are blocked
  • Centralized security control

One-line Summary

I configure a Lambda Authorizer in API Gateway to validate tokens and enforce authorization before the request reaches the business Lambda.
